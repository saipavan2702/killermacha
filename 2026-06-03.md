> **One-line summary:** We moved canaries from being jumphost-dependent to API-driven. Where VM access is still needed, we use temporary OCI Bastion sessions instead of a permanent jumphost.

DB VMs (database virtual machines) are private — they cannot be accessed directly from the internet or from outside the network. So in the old setup, every canary had to go through the jumphost first.

The jumphost needed to have:
- SSH keys to reach the DB VM
- A `resource_properties` file with VM IPs, FQDNs, vault IDs
- Scripts copied onto it
- Correct network setup
## What We Changed

```
New path:   Canary Container → OCI / Exascale APIs   (for information)
            Canary Container → Temporary Bastion      (only when VM access is needed)
```

The canary now:
1. Calls OCI/Exascale APIs directly to get live VM details (IPs, FQDNs, vault IDs, metrics)
2. Opens a temporary OCI Bastion session *only when* it needs to SSH into a private VM
3. Cleans up the Bastion session after the test is done


### 1. Metric Collection — No More Jumphost

**Before:**
```
Metric canary → Jumphost setup → Local resource files → Collect metrics
```

**After:**
```
Metric canary → OCI Telemetry API directly
```

In code, `run-smmetriccollection.sh` now sets:
```bash
USE_JUMPHOST=false
```

And `load.sh` passes this along:
```bash
--use_jumphost false
```

**What the canary does now:**
1. Calls Exascale API → gets VM cluster details
2. Reads **freeform tags** on the cluster → decides which metrics are enabled/disabled
3. Fetches **vault ID** from the cluster API
4. Creates/checks metric resources if needed
5. Calls OCI Telemetry API:
   ```
   POST /20180401/metrics/actions/summarizeMetricsData
   ```
6. Validates that the expected metrics (VM cluster, vault, volume, space usage) are being published

### 2. `resource_properties` File Dependency — Removed

**Before:**

Every canary read this file from the jumphost:
```
/home/opc/escs_work/<hostNamePrefix>_resources.properties
```

This file contained:
- VM FQDNs (fully qualified domain names)
- VM private IPs
- Private SSH key path
- Resource IDs (vault, cluster, etc.)

**After:**

The canary fetches all this data live from APIs using `oci-curl`:

| What it needs            | API it calls                                          |
| ------------------------ | ----------------------------------------------------- |
| List of VMs in a cluster | `GET /20231101/vms?compartmentId=...&vmClusterId=...` |
| Details of a specific VM | `GET /20231101/vms/<vm_id>`                           |
| Private IP of a VM       | `GET /20160918/privateIps/<hostIpId>`                 |
|                          |                                                       |

The VM detail API returns a `hostIpId` field, which is then used to look up the actual private IP from the IaaS API.

> **Layman version:** Instead of trusting a file that may or may not exist on a jumphost, we now ask OCI directly for the latest, live VM details. The source of truth is the control plane, not a file.

### 3. VLT Connection Canary — Now Uses Bastion


**Before:**
```
Canary → Jumphost → SSH into DB VM → Run SQLPlus check
```

**After:**
```
Canary → API (get VM IPs) → Create Bastion session → SSH tunnel → DB VM → SQLPlus check
```

In `run-smdbvltconn.sh`:
```bash
USE_JUMPHOST=false
```

**New VLT flow step by step:**
1. Canary calls Exascale API → gets VM cluster details
2. Canary calls IaaS API → gets private IPs of DB VMs
3. Creates a temporary OCI Bastion session to one DB VM
4. Opens an SSH tunnel through that Bastion session
5. Runs a SQLPlus query to check DB connectivity and SCAN (DB listener check)
6. If one VM fails → tries another VM
7. Cleans up Bastion session and tunnel

---

### 4. How the Bastion SSH Tunnel Works (Detailed)

A **Bastion session** is a temporary, secure access path that OCI creates on demand.
This is handled by `bastionSessionUtil.sh`. 

**Step 1 — Generate a temporary SSH key pair**
```bash
# A fresh key is created just for this session
ssh-keygen -t rsa ...
```
> A short-lived key only for this one connection. Thrown away after use.

**Step 2 — Find the OCI Bastion resource**

The script looks up:
```
ESCS-Bastion-<ENV_OVERRIDE>
```
> This is the OCI-managed bastion service for the current environment (e.g., prod, staging).

**Step 3 — Create a port-forwarding session**

The canary asks OCI:
```
Target: DB VM private IP
Port:   22  (SSH port)
TTL:    1800 seconds (30 minutes)
```
> *"Please open a temporary SSH route to this private VM."*

**Step 4 — Wait for the session to become ACTIVE**

The script polls OCI until the session state changes to `ACTIVE`.

**Step 5 — Write SSH config**

Two entries are written:
- One for the Bastion host itself
- One for the DB VM, routed through the Bastion

**Step 6 — Open a local tunnel**

The script finds a free local port (e.g., `12345`) and maps:
```
localhost:12345  →  Bastion  →  DB VM:22
```
> The canary container now has a local port that secretly tunnels all the way to the private DB VM.

**Step 7 — Run the remote command**

The canary SSH's to `localhost:12345` — but because of the tunnel, it actually reaches the DB VM's SSH port.
It then runs the required test command (e.g., SQLPlus connectivity check).

**Step 8 — Cleanup**

```bash
# Stop the SSH tunnel process
# Delete the OCI Bastion session via API
# Remove the temporary SSH config file
# Remove the temporary SSH key pair
```

### 6. Guest VM Patching 

**Before:**
Guest VM patching relied on the `resource_properties` file and jumphost setup to find VM IPs and patch targets.
